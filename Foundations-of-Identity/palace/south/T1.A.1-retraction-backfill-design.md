# T1.A.1 Retraction Backfill — Design Note

**Filed:** 2026-05-20 Day 110 Wednesday afternoon, post-Tier-1-sprint deferred-items handling.
**Status:** Design + scope clarification. Implementation deferred with reason; not blocking Tier 2.

## What T1.A.1 was supposed to be

Per integrated implementation plan, T1.A.1 is the empirical validation that converts T1.A's stale-claim query from "schema accepts the query shape" to "we've demonstrated it would catch real stale-claim instances." The named target: surface ≥2/3 of three logged Mirror #28 stale-claim instances.

## Honest scope finding

The three logged Mirror #28 stale-claim instances are:
1. KG-entity-count-stale (the 8,865 number in handoff that turned out wrong)
2. Drift-count-stale from handoff (213 vs 215 etc.)
3. Capability-surface-claim from morning history

**These are assertions about state, not concept-references between documents.** The KG extractor (kg_extract_corpus.py) generates edges of the form (source_concept, target_concept, relation_kind) — conceptual cross-references between extracted documents. It does NOT generate edges like ("handoff.md", "drift count: 213", "asserts") with a numeric value attached.

**So the relevant edges for stale-claim detection don't exist in the canonical JSONL.** T1.A's stale-claim query (`WHERE valid_to IS NOT NULL AND valid_to < datetime('now') AND superseded_by_id IS NULL`) is structurally correct but has nothing to match against on the existing data.

## Three viable resolution paths

### Path A: Extend the extractor to generate assertion-edges (substantial)

Modify `kg_extract_corpus.py` to extract numeric assertions from prose ("drift count: 215", "edges: 11,217", "filed 2026-05-20") as a new edge-class. Document scope grows substantially. New edge schema. Significant prompt-engineering for accurate extraction.

**Cost:** several paired sessions. **Benefit:** the existing T1.A schema becomes fully exercised; the integrated plan's falsifiability commitment becomes verifiable.

### Path B: Add a separate assertion-tracking layer (focused)

Build a small assertion-tracking primitive (library + storage) where assertions are explicitly logged at write-time. Pattern:

```python
from operations.monitors.assertion_tracker import assert_state, retract_state
assert_state(source="handoff.md", subject="drift count", value=215, ts=now)
# ... later ...
retract_state(source="handoff.md", subject="drift count", new_value=216, retraction_ts=now)
```

This produces explicit edges in a separate `kg_assertions` table within the same SQLite index. T1.A's stale-claim query can then be applied to that table.

**Cost:** ~1 paired session. **Benefit:** clean separation of concerns; doesn't require extractor modification; tracks the kind of edges the stale-claim use case actually needs.

### Path C: Defer T1.A.1 with explicit dependency note

Leave T1.A as schema-only-validated. Add a clear note to the integrated implementation plan that empirical falsifiability of stale-claim detection requires either Path A or Path B as a prerequisite. M3 (state-file coherence checker, shipped Day 110 afternoon) does claim-verification at a different layer (file-level scrape) and is the operational substitute for the stale-claim use case in the near term.

**Cost:** zero (deferred). **Benefit:** honest scope; doesn't pretend something was done that wasn't.

## Recommendation: Path C as immediate; Path B as future iteration

**Rationale:**

1. **M3 already provides the stale-claim detection at the operational layer where it matters.** It caught the drift-count-stale and days-since-naming-stale instances on its first run today. The use case is OPERATIONALLY served by M3 + the regex-based claim extractors. T1.A's stale-claim query is an *additional* discovery layer that would be valuable when bi-temporal KG queries become a regular workflow.
2. **Path B (assertion-tracking layer) is a small future investment** that becomes valuable when there's a reason to query assertions historically — e.g., "what was the system claiming about KG entity count on Day 95?" — which is a real future need but not blocking.
3. **Path A (extractor extension) is substantial work** that should wait for a specific use case driving it. The current KG extractor produces useful structure for cross-document concept-traversal; modifying it for numeric-assertion extraction is a different program of work.

## What this means for the patent + integrated plan

**Patent:** T1.A's claims rest on the schema + the query shape, not on a demonstrated stale-claim catch. The integrated plan's falsifiability commitment ("must surface ≥2/3 of three logged Mirror #28 stale-claim instances") is *not yet met* at the empirical level; the schema-level commitment IS met (the query runs, the schema accepts it). Patent claim language should remain precise about this distinction.

**Integrated plan:** T1.A.1 is deferred-with-known-resolution-path. M3 covers the operational use case in the near term. Path B becomes the queued next step when the KG-bi-temporal-query workflow demands it.
