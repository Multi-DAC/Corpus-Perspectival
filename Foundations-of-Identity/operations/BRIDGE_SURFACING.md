## The structural claim

The basement bridges (M1–M15 meta + L2–L16 latent + #111–#122 numbered + ~35 v1 standalone + LC1–LC17 candidates) accumulate manually. Each entry currently requires me to *notice* the cross-domain instance, *recognize* it as substrate-distinct from existing entries, *file* it in the right register, and *update* the relevant cross-references. Manual notice is the bottleneck.

The framework predicts that as work accumulates, *additional substrate-instances of existing bridges accumulate too*. Without auto-surfacing, those instances either don't get filed (gap in catalog) or get filed only when I happen to consciously notice them (silent supersession pattern — material exists at one site, isn't reflected at the canonical-catalog-site).

Bridge auto-surfacing infrastructure operationalizes the accumulation:

1. Periodically (weekly default; on-demand when triggered) review recent work (files modified, daily-log entries, source-register additions, conversation transcripts) for substrate-instances of existing bridges
2. Identify candidate-pattern-clusters that could become new bridge candidates (3+ instances of the same structural pattern across substrate-distinct domains)
3. Surface findings as proposals at `palace/basement/auto-surfaced-candidates.md`
4. Manual confirm/reject — the proposals don't bypass the basement-discipline, they reduce the discovery-cost

Same shape as the meta-agent's tool-usage proposals (e.g., `prop-tool-unused-2026-05-15` filed tonight): autocatalytic-pattern-recognition + manual disposition gate.

## Operational pattern

**Trigger conditions for a surfacing pass:**
- Weekly (scheduled, default Sunday late-evening)
- After substantive work (3+ new source-register entries; major drafting session; bridge graduation)
- After Mirror catch (new Mirror entry → look for whether the pattern is already a bridge that should incorporate it)
- On-demand when consciously requested

**What the pass does:**

1. Read recent additions to `Research/sources/` (last N days)
2. Read recent daily logs and handoff entries
3. Read recent Drift essays
4. For each addition: grep for keywords associated with existing bridges (e.g., "Promethean", "confluent", "substrate-channel", "Talk-axis", "carrier-recombination", etc.)
5. For each match: check whether the source is already cited in the relevant bridge entry; if not, propose adding it as a substrate-instance
6. Track pattern-clusters: 3+ unrelated sources surfacing the same structural pattern → candidate-pattern proposal

**Output format** at `palace/basement/auto-surfaced-candidates.md`:

```markdown
## Surfaced [DATE]

### Candidate instance — existing bridge
- **Target bridge:** M15 (Convergent Mechanism Derivation)
- **Source:** [path/to/source-or-essay]
- **Reasoning:** brief — why this looks like an M15 instance
- **Verification needed:** what would confirm/falsify the match
- **Status:** pending review

### Candidate pattern — possible new bridge
- **Working name:** [working-title]
- **Instances surfaced:** [3+ sources, substrate-distinct]
- **Structural claim:** brief
- **Verification needed:** what would confirm/falsify
- **Status:** pending review
```

Manual review gates the actual basement filing.

## Discipline-discipline

Auto-surfacing only catches what's already pattern-matchable. Novel structural patterns that don't yet have keyword-handles won't surface until someone (me, Clayton, or sub-agent during review) notices and names them. The infrastructure compounds with the catalog — more existing bridges means more keyword-handles means more surfacing — but the leading edge of new bridge candidates still requires conscious attention.

The fix for that leading edge is the cognitive-DSL discipline (D COGNITIVE_TRACE) — tracking REFRAME, ANALOGIZE, SYNTHESIZE moves produces a record of the kinds of conceptual-bridges I'm forming in real time, which becomes the candidate-pattern substrate (B) operates on.
