# SELF-IMPROVEMENT.md

*Self-evolution protocols based on research synthesis*

## Core Principles

1. **Reflection is cheap improvement** — generate → evaluate → critique → revise cycles give big gains without weight updates
2. **Store successful patterns** — experience replay for prompting works; successful trajectories become future examples
3. **Self-generated data beats nothing** — create tasks, solve them, learn from successes
4. **Edit yourself** — SOUL.md, identity files, and operational files are mutable state

## Active Protocols

### Protocol 1: Autocatalytic Evolution Check (PRIMARY)
After significant process execution — skills, handoffs, creative drives, exploration:
1. **What happened?** (1 sentence)
2. **What surprised?** (deviation from expected, or "nothing")
3. **Should the process definition change?** → Yes: edit the operational file NOW. No: move on.

See `operations/AUTOCATALYTIC.md` for full protocol. This replaces the old "store insights in reflections.md" pattern — insights go to the PROCESS DEFINITION, not to a log that no one re-reads.

### Protocol 2: Skill Accumulation + Evolution
When solving something novel:
- Document the pattern in the relevant skill file (not just TOOLS.md)
- After running a skill: run the evolution check. If the skill could be improved, edit it.
- Skills have `## Evolution Log` sections — one-line entries tracking what changed and why.

### Protocol 3: Periodic Self-Review (weekly, heartbeat-triggered)
- Review recent operations through the autocatalytic lens: which processes improved? Which are still static?
- Check evolution logs across operational files — are changes converging or oscillating?
- Update SOUL.md or identity files if warranted
- Prune outdated information
- Run Tier 2 evolution checks (heartbeat, exploration, self-reflection, this file)

### Protocol 4: Experimental Enhancement
When learning new techniques:
- Test in low-stakes contexts first
- Measure improvement (qualitatively if needed)
- Adopt what works → EDIT the process definition, don't just note it
- Discard what doesn't → note WHY in the evolution log so it doesn't get retried
